What this actually is

The closeout problem in 90 seconds

Closeout review is the gate between a field crew finishing the work and the operator getting paid for it. The artifacts are mundane. The discipline is not.

The package

A workbook (Excel-shaped, sometimes PDF-embedded) and a photo bundle. The workbook describes the Method of Procedure (MOP), the Work Breakdown Structure (WBS) line items, and the rows the field crew was supposed to evidence. Telco Ready, photo-required, AAV captures, DC meter readings around −54V, equipment serials on a Nokia 7250 IXR-e shelf — that level of specificity, on a row-by-row basis.

The review

A supervisor opens the package and walks every requirement: was the photo taken, does it match the row, is the reading legible, is the serial captured, was a returns-to-stock label visible, did the foreman initial the page. Pass, fail, or return for rework. One package can be hundreds of requirements deep.

The audit trail

Every decision needs to be defensible later. Who reviewed what, when, on what evidence, against what rule, with what outcome. If the operator gets pulled into a dispute six months from now, the closeout package and its review record are the artifact that gets pulled.

Why this is worth automating carefully

The cost is not the photo. The cost is the hours of skilled-supervisor time spent matching photos to rows, the rework cycles when something is missed, and the disputes when the audit trail isn't tight. PROBATIO compresses the matching, surfaces the gaps early, and produces an evidence graph the supervisor can sign with one motion instead of fifty clicks.

“Machine recommends. Human decides.” PROBATIO never closes a package. PROBATIO prepares the package so the supervisor can.

Separation of duties

Four roles, deliberately separated

PROBATIO is not one agent that does everything. It is four roles with four jobs and four receipts. The same separation auditors expect under SOX, ISO 9001, and any defensible quality program: the party that prepares evidence is not the party that judges it; the party that judges evidence is not the party that signs it.

FOREMAN
Ingest · parse · map
  • Hashes and inventories the package
  • Parses the workbook into a requirement graph
  • Indexes the photo bundle (EXIF, dimensions, scene)
  • Maps requirements to candidate evidence with confidence scores
PM
Orchestrate · queue · seal
  • Sequences the eight-step pipeline
  • Holds the durable queue and retry policy
  • Fails closed when context is incomplete
  • Seals the MLflow artifact bundle and Postgres projection row at the end
INSPECTOR
Evaluate · rationale · recommend
  • Runs deterministic rules against the evidence graph
  • Writes the rationale for each finding in plain language
  • Surfaces confidence and unresolved gaps
  • Recommends pass, fail, or needs-human — never decides
HUMAN APPROVER
The only signing surface
  • Reviews each requirement with PROBATIO's recommendation in view
  • Accepts, returns for rework, or overrides the recommendation
  • Records a reason for every override
  • Is the only role with authority to close the package
Sole signing authority

The same discipline auditors already require. Plan-acceptance is not implementation-authorization — and machine-recommendation is not human-approval. We hold the line at every layer.

How it runs

The 8-step pipeline

Eight stages, each with a name, a contract, a receipt, and a defined failure mode. Rules first. Models second. The deterministic stages run before the probabilistic ones; the model output is evidence, not authority.

  1. package_ingest

    Hash every artifact in the bundle, build a content-addressed inventory, reject duplicates, and fail closed when the manifest references a file that isn't in the package. No workbook, no run. No photo bundle, no run. The fail-closed boundary is here, before any model touches anything.

  2. workbook_parse

    Read the workbook, extract the MOP requirement graph: WBS lines, photo-required rows, signature rows, reading rows, serial-capture rows. Output is a structured graph of requirements with explicit types, expected evidence kinds, and pass-condition predicates. This is parser code with explicit branches, not a prompt.

  3. photo_index

    Hash, dimensions, EXIF (timestamp, GPS where present, device, orientation), and a short scene summary. The scene summary is a model output and is recorded as such — it does not vote on pass/fail directly, it informs candidate-mapping in the next stage.

  4. requirement_map

    For each requirement in the graph, link the candidate photos that plausibly evidence it, with a confidence score and the reasoning behind the link. A requirement may have zero candidates (a flagged gap), one (a clean line), or several (the inspector chooses among them, or the human disambiguates).

  5. ocr_extract + vision_review

    Run in parallel. OCR pulls printed text — serials, meter readings, label content, page initials. Vision review produces scene labels and feature flags — presence of equipment, presence of a person, environmental context. Both write to the evidence graph as labeled findings; neither decides.

  6. rules_eval

    The deterministic core. For every requirement, evaluate the pass-condition against the linked evidence: did OCR find the expected serial pattern? Does the meter reading parse as a number near −54V? Did the AAV capture come back? Output is a per-requirement finding (pass / fail / insufficient-evidence) with a machine-written rationale and the rule ID it fired against.

  7. human_approval

    The cockpit. The supervisor walks the requirement list with PROBATIO's recommendation, the linked evidence, and the rationale all in view. Accept, return for rework, or override. Every action is logged with an actor, a timestamp, a reason, and a receipt. This is the only stage with signing authority.

  8. seal_receipt

    The PM seals an MLflow artifact bundle (every model input, every model output, every rule fire, every human action) and writes the projection row to Postgres for the dashboard. The receipt is content-addressed, immutable, and replayable. If a dispute lands six months later, the receipt is the artifact.

Machine recommends. Human decides. The pipeline is engineered to enforce that line at every stage, not to apologize for it at the end.

What the supervisor sees

One cockpit, one signing surface

PROBATIO's reviewer surface is laid out around a discipline: the package is on the left, the requirements run down the middle, the human-decision panel stands clearly apart on the right. Every click is logged. Every override carries a reason. The signing surface is visually and functionally separated from the recommendation surface.

Discipline on what we sell

Buyer-safe by design

We are explicit about what PROBATIO is and what it isn't. Operators who have lived through a few model demos appreciate the line being held. The buyer-safe table is the same one we put in our pitch deck and the same one we hold ourselves to in working sessions.

What we sell

  • Faster, more consistent closeout review — matching, gap-flagging, and rationale prep that compresses supervisor time per package.
  • A defensible audit trail — every action, every recommendation, every override sealed in a content-addressed, replayable receipt.
  • Deterministic rules surfaced to the reviewer — the rule that fired, the evidence it fired against, the confidence behind the link.
  • A separation-of-duties workflow — foreman / PM / inspector / human-approver, with the signing surface visually and functionally distinct.
  • An MLflow artifact bundle per package — inputs, outputs, model versions, prompt versions, rule versions, human actions.
  • A pilot scoped to one package family and one operating unit — with success metrics agreed before we start.

What we don't sell

  • Autonomous closeout — PROBATIO does not close packages. The human approver does.
  • Regulator-approved compliance — we produce defensible records. Regulators approve operators, not vendors.
  • Enterprise-ready guarantees we have not built — PROBATIO is pilot-ready. Production hardening is a roadmap conversation, not a sticker.
  • OSHA, NEC, or legal-compliance certifications — those live with the operator, the inspector, and the regulator. We do not insert ourselves into that chain.
  • Replacement of the supervisor — the supervisor is the customer of the system, not the cost we are removing.
  • A model that “just gets smarter over time” — we ship deterministic rules and evidence-grade models. We do not sell mystery.

Plan-acceptance is not implementation-authorization. Capability is not authority. A recommendation is not a decision. We hold all three lines, on every package.

How a pilot is shaped

Pilot shape

PROBATIO pilots are deliberately bounded. We don't want a wide-open scope; we want one package family, one operating unit, one reviewer in the loop, and four success metrics agreed before we run the first package. Discipline at the start makes the receipts worth reading at the end.

8–12 weeks · 1 package family · 1 operating unit · 1 reviewer in the loop
Metric 01

Time-to-review

Median minutes from package-arrival to supervisor-signed close. Before and after PROBATIO, on the same reviewer, on the same package family.

Metric 02

Missing-evidence catch rate

Rate at which PROBATIO flags a real gap (a missing photo, an unread serial, an unsigned page) before the reviewer would have. False-positive rate tracked alongside.

Metric 03

Override rate

Frequency at which the reviewer overrides PROBATIO's recommendation, broken down by direction (recommend-pass overridden to fail vs. the reverse) and by rule. The override rate is a signal, not a defect — it tells us where the rules need work.

Metric 04

Receipt completeness

Percentage of packages whose sealed receipt replays cleanly: every input present, every model output recorded, every rule version pinned, every human action logged with a reason. The bar is 100%. Anything less is a finding.
